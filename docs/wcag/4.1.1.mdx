# WCAG-succescriterium 4.1.1 Parsen

**Let op**: Dit succescriterium is in WCAG-versie 2.2 verwijderd. Pas wanneer de opvolger van [EN_301_549 3.2.1](https://nl.wikipedia.org/wiki/EN_301_549) overstapt van WCAG 2.1 naar 2.2 dan is dit succescriterium niet meer wettelijk verplicht.

## Uitleg

Voor WCAG 2.1 moet de "markup parsen" geen problemen opleveren, dat betekent dat de HTML-code foutloos moet zijn. Dan werkt een webpagina beter.

Wat betekent parsen eigenlijk? Officieel: het analyseren van informatie aan de hand van losse componenten, of het analyseren van de syntax.

Er zijn 4 voorwaarden gedefinieerd.

### 1. Elementen zijn volgens hun specificaties afgesloten

Denk hier bijvoorbeeld aan paragrafen, headings, divs, spans en andere elementen die een sluittag vereisen.
Een `<div>` moet afsluiten met een `</div>`.

### 2. Elementen zijn genest volgens hun specificaties

Verkeerd nesten van elementen kan voorkomen bij interactieve elementen: een button die een tweede button bevat.
Of bij het verkeerd combineren van elementen: een label die een link bevat.
Daarnaast moeten bijvoorbeeld lists en definition lists goed opgebouwd zijn.

Silvestar BistroviÄ‡ beschrijft een aantal [<span lang="en">Common nesting issues in HTML</span>](https://www.htmhell.dev/adventcalendar/2022/20/).

### 3. Elementen bevatten geen dubbele attributen

Combineer waardes in attributen en herhaal de attributen zelf niet.
Bijvoorbeeld voor het toekennen van CSS-classes:

```html
<!-- Foute code, niet gebruiken -->
<h1 class="heading" class="x-large">Contact</h1>
```

Voeg de classes samen, gescheiden door een spatie:

```html
<h1 class="heading x-large">Contact</h1>
```

### 4. Alle ID's zijn uniek, behalve waar de specificatie deze eigenschappen toelaat

Dit is voor formulieren belangrijk. Labels en formuliervelden zijn via een ID aan elkaar gekoppeld. Komt een ID meerdere keren voor op een pagina, dan kan het verkeerde label bij een formulierveld worden voorgelezen.

In code:

```html
<label for="kleur">Wat is je lievelingskleur?</label> <input type="text" id="kleur" name="lievelingskleur" />
```

Bij de richtlijnen over labels staat hoe deze [for/id relatie](/richtlijnen/formulieren/labels/de-forid-relatie) werkt.

Een ander probleem is het toevoegen van twee keer hetzelfde formulier met hetzelfde formulier-ID, de foutafhandeling staat dan misschien bij het verkeerde formulier.

## Hoe te testen

- Controleer de webpagina op de [Markup Validation Service](https://validator.w3.org/).
- Controleer de webpagina met de [span lang="en">WCAG 2.1 parsing error bookmarklet</span>](https://www.tpgi.com/wcag-2-0-parsing-error-bookmarklet/) van Steve Faulkner
- [axe Devtools](https://www.deque.com/axe/browser-extensions/) markeert enkele (maar niet alle) parse-fouten in de HTML.

Voor developers:

- Gebruik een [HTML-linter](https://github.com/caramelomartins/awesome-linters?tab=readme-ov-file#html), of een linter naar keuze die ook voor correcte HTML test.
- Bij twijfel: check de '<span lang="en">Permitted content</span>' bij de '<span lang="en">Technical Summary</span>' van een HTML-element op de MDN-website. Bijvoorbeeld [de technische samenvatting voor een link](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/a#technical_summary).

## Veelgemaakte fouten

### Fout: meerdere keren hetzelfde Gravity Forms formulier op een webpagina.

De WordPress formulieren-plugin [Gravity Forms](https://docs.gravityforms.com/accessibility-checklist-for-gravity-forms/) geeft elk formulier een eigen ID. Voeg je hetzelfde formulier twee keer toe op een pagina, dan worden de fouten afgehandeld bij het bovenste formulier. Dit is verwarrend als je net het onderste formulier aan het invullen bent.
Een formulier dat vaak twee keer wordt toegevoegd, is een aanmeldformulier voor de nieuwsbrief. Wil je toch hetzelfde formulier meerdere keren op een pagina, maak dan verschillende formulieren hiervoor aan.
