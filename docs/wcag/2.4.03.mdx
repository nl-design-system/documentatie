## Uitleg

<Summary />

Met de Tab-toets verplaatst de focus zich naar het volgende interactieve element. Met Shift+Tab gaat de focus terug. De focusvolgorde hoeft niet exact overeen te komen met de visuele weergave, maar mag niet zodanig afwijken dat het verwarrend wordt.

De focusvolgorde wordt bepaald door de volgorde van interactieve elementen in de HTML. Je kunt die volgorde aanpassen met het `tabindex`-attribuut of met JavaScript. Pas daar wel mee op: wijzigingen hierin kunnen je website ontoegankelijk maken en je moet je werk zorgvuldig testen op alle resoluties.

### Gerelateerde succescriteria

SC 2.4.3 richt zich specifiek op de volgorde van focusbare elementen. Dit criterium raakt aan meerdere andere SC's:

- **SC 1.3.2** — De leesvolgorde van content.
- **SC 1.4.11** — De focusindicator moet voldoende contrast hebben.
- **SC 2.4.7** — De zichtbaarheid van de toetsenbordfocus.
- **SC 2.4.11** — De focus mag niet door andere elementen worden bedekt.
- **SC 3.2.1** — De focus mag niet automatisch verplaatsen bij interactie met een element (bijvoorbeeld het aanvinken van een checkbox).
- **SC 3.2.2** — Er mogen geen onverwachte wijzigingen optreden als een element focus krijgt.

## Veelgemaakte fouten

In de onderstaande beschrijvingen wordt gesproken over dialoogvensters (modal dialogs): elementen die boven de webpagina worden getoond en de interactie met de onderliggende pagina blokkeren, zoals lightboxes, mobiele menu's, cookiebanners en videomodals.

### Fout: chaotische focusvolgorde

De focus springt onvoorspelbaar tussen verschillende secties zonder logisch patroon. Dit gebeurt vaak als de visuele volgorde (via CSS) niet overeenkomt met de volgorde in de HTML. Bijvoorbeeld: de HTML begint met een sidebar, maar visueel staat die rechts. De focus springt dan eerst door de sidebar voordat het naar de hoofdcontent gaat.

### Fout: focus wordt niet naar een dialoogvenster verplaatst

Als een dialoogvenster de pagina bedekt maar de focus niet naar dat venster verplaatst, kan de bezoeker het venster niet bedienen met het toetsenbord. De focus loopt over de interactieve elementen onder het venster.

```javascript
// Fout: dialoogvenster openen zonder focus te verplaatsen
function openDialog() {
  document.getElementById("dialog").style.display = "block";
  document.getElementById("overlay").style.display = "block";
}

// Goed: focus verplaatsen naar het dialoogvenster
function openDialog() {
  const dialog = document.getElementById("dialog");
  dialog.style.display = "block";
  document.getElementById("overlay").style.display = "block";
  dialog.querySelector("button, [href], input")?.focus();
}
```

### Fout: focus verlaat het dialoogvenster terwijl het nog open is

Na het doorlopen van alle interactieve elementen in een dialoogvenster (zoals een lightbox of mobiel menu) kan de focus het venster verlaten terwijl het nog open is. De bezoeker navigeert dan door elementen achter het venster. Als het venster bovendien niet met Escape te sluiten is, weet de bezoeker niet waar de focus is.

```javascript
// Goed: focus trap in een dialoogvenster
function trapFocus(dialog) {
  const focusableElements = dialog.querySelectorAll(
    'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])',
  );
  const first = focusableElements[0];
  const last = focusableElements[focusableElements.length - 1];

  dialog.addEventListener("keydown", (e) => {
    if (e.key === "Escape") {
      sluitDialog();
      return;
    }
    if (e.key !== "Tab") return;

    if (e.shiftKey && document.activeElement === first) {
      e.preventDefault();
      last.focus();
    } else if (!e.shiftKey && document.activeElement === last) {
      e.preventDefault();
      first.focus();
    }
  });
}
```

### Fout: focus keert niet terug naar het verwachte element

Na het sluiten van een dialoogvenster keert de focus niet terug naar de knop die het venster heeft geopend of naar een ander element naast deze knop. De bezoeker is de draad kwijt en moet opnieuw door de pagina tabben om te vinden waar die was.

```javascript
// Fout: focus verdwijnt na sluiten
function sluitDialog() {
  document.getElementById("dialog").style.display = "none";
  // Focus gaat nergens naartoe — belandt op <body>
}

// Goed: focus terug naar de trigger
let triggerElement = null;

function openDialog(trigger) {
  triggerElement = trigger;
  document.getElementById("dialog").style.display = "block";
  document.getElementById("dialog").querySelector("button")?.focus();
}

function sluitDialog() {
  document.getElementById("dialog").style.display = "none";
  triggerElement?.focus(); // Terug naar de knop die het venster opende
}
```

### Fout: cookiebanner krijgt niet als eerste de focus

Een cookiebanner verschijnt bij het laden van de pagina, maar staat onderaan in de HTML en daardoor onderaan in de focusvolgorde. De banner bedekt de inhoud van de pagina, maar de bezoeker moet eerst door de volledige pagina tabben om de banner te bereiken.

```html
<!-- Fout: cookiebanner onderaan de HTML; focus NIET naar de banner -->
<header>...</header>
<main>...</main>
<footer>...</footer>
<div class="cookiebanner">
  <p>Wij gebruiken cookies.</p>
  <button>Accepteren</button>
  <button>Weigeren</button>
</div>

<!-- Goed: cookiebanner bovenaan de HTML of met focus management -->
<div class="cookiebanner" role="dialog" aria-label="Cookiemelding">
  <p>Wij gebruiken cookies.</p>
  <button autofocus>Accepteren</button>
  <button>Weigeren</button>
</div>
<header>...</header>
<main>...</main>
<footer>...</footer>
```

### Fout: focusvolgorde in een accordion of tabs is onjuist

In een accordion moeten de knoppen die panelen openen achtereenvolgens focus krijgen. Als een paneel open is en interactieve elementen bevat, moet de focus naar die elementen gaan voordat de volgende knop aan de beurt is. Dit probleem komt ook voor bij "Meer opties"-knoppen in filters of op overzichtspagina's.

```html
<!-- Fout: geopend paneel staat na alle knoppen in de HTML -->
<button aria-expanded="true">Sectie 1</button>
<button aria-expanded="false">Sectie 2</button>
<button aria-expanded="false">Sectie 3</button>
<div class="paneel-1">
  <a href="/link-in-paneel-1">Link in paneel 1</a>
</div>

<!-- Goed: paneel staat direct na de bijbehorende knop -->
<button aria-expanded="true" aria-controls="paneel-1">Sectie 1</button>
<div id="paneel-1" role="region">
  <a href="/link-in-paneel-1">Link in paneel 1</a>
</div>
<button aria-expanded="false" aria-controls="paneel-2">Sectie 2</button>
<div id="paneel-2" role="region" hidden>...</div>
<button aria-expanded="false" aria-controls="paneel-3">Sectie 3</button>
<div id="paneel-3" role="region" hidden>...</div>
```

### Fout: focusvolgorde in een carrousel is onjuist

De focus doorloopt de navigatieknoppen van een carrousel correct, maar als een slide interactieve elementen bevat (links, knoppen), wordt de focus niet naar die elementen verplaatst. De bezoeker kan de inhoud van de actieve slide niet bereiken.

```html
<!-- Fout: slide-content is niet focusbaar -->
<div class="carrousel">
  <button aria-label="Vorige slide">&#8249;</button>
  <button aria-label="Volgende slide">&#8250;</button>
  <div class="slide actief">
    <h3>Aanbieding</h3>
    <a href="/aanbieding" tabindex="-1">Bekijk aanbieding</a>
    <!-- tabindex="-1" haalt de link uit de focusvolgorde -->
  </div>
</div>

<!-- Goed: interactieve elementen in de actieve slide zijn focusbaar -->
<div class="carrousel">
  <div class="slide actief">
    <h3>Aanbieding</h3>
    <a href="/aanbieding">Bekijk aanbieding</a>
  </div>
  <button aria-label="Vorige slide">&#8249;</button>
  <button aria-label="Volgende slide">&#8250;</button>
</div>
```

### Fout: focus wordt niet naar een foutmelding verplaatst

Na het verzenden van een formulier verschijnen foutmeldingen bovenaan het formulier, maar de focus blijft op de verzendknop staan. De bezoeker merkt niet dat er fouten zijn en weet niet waar die staan.

```javascript
// Fout: foutmeldingen tonen zonder focus te verplaatsen
function valideerFormulier() {
  const fouten = vindFouten();
  if (fouten.length > 0) {
    toonFoutmeldingen(fouten);
    // Focus blijft op de verzendknop
  }
}

// Goed: focus naar de eerste foutmelding
function valideerFormulier() {
  const fouten = vindFouten();
  if (fouten.length > 0) {
    toonFoutmeldingen(fouten);
    document.querySelector(".foutmelding")?.focus();
  }
}
```

### Fout: skiplink verplaatst de focus niet naar het juiste doel

Een skiplink moet de focus direct naar de unieke hoofdcontent van de pagina brengen. Als de skiplink alleen de pagina scrollt maar de focus niet verplaatst, navigeert de bezoeker na de volgende Tab-druk weer vanaf het begin van de pagina.

```html
<!-- Fout: skiplink zonder focusbaar doel -->
<a href="#main">Ga naar inhoud</a>
...
<main id="content">
  <h1>Paginatitel</h1>
</main>

<!-- Goed: doel is focusbaar -->
<!-- In sommige browsers krijgt main geen focus zonder tabindex -->
<a href="#main">Ga naar inhoud</a>
...
<main id="main" tabindex="-1">
  <h1>Paginatitel</h1>
</main>
```

### Fout: in-page links verplaatsen de focus niet

Een pagina bevat een inhoudsopgave. Na het klikken op een link scrollt de pagina naar de juiste sectie, maar de toetsenbordfocus blijft op de link in de inhoudsopgave staan. Bij de volgende Tab-druk spring je terug naar boven. Dit probleem komt ook voor bij "scroll naar boven"-links.

```html
<!-- Fout: heading is niet focusbaar -->
<nav>
  <a href="#sectie-2">Hoofdstuk 2</a>
</nav>
...
<h2 id="sectie-2">Hoofdstuk 2</h2>

<!-- Goed: heading is focusbaar met tabindex="-1" -->
<nav>
  <a href="#sectie-2">Hoofdstuk 2</a>
</nav>
...
<h2 id="sectie-2" tabindex="-1">Hoofdstuk 2</h2>
```

### Fout: positieve tabindex maakt focusvolgorde onlogisch

Een developer heeft `tabindex`-attributen met positieve waarden toegevoegd om de focus binnen een widget te sturen. Het resultaat: de focusvolgorde op de volledige pagina is onlogisch en onvoorspelbaar. Elementen met een positieve `tabindex` krijgen altijd als eerste focus, ongeacht hun positie in de HTML.

```html
<!-- Fout: positieve tabindex verstoort de hele pagina -->
<nav>
  <a href="/">Home</a>
  <a href="/over-ons">Over ons</a>
</nav>
<main>
  <input tabindex="2" type="text" placeholder="Zoeken" />
  <button tabindex="1">Verzenden</button>
  <!-- Focus gaat eerst naar Verzenden (1), dan Zoeken (2),
       dan pas de navigatielinks (0) -->
</main>

<!-- Goed: gebruik de HTML-volgorde, geen positieve tabindex -->
<nav>
  <a href="/">Home</a>
  <a href="/over-ons">Over ons</a>
</nav>
<main>
  <input type="text" placeholder="Zoeken" />
  <button>Verzenden</button>
</main>
```

## Hoe te testen

### Voor iedereen

1. Klik in de adresbalk van de browser.
2. Navigeer met de **Tab**-toets van het ene interactieve element naar het andere. Volgt de focus een logische volgorde?
3. Navigeer terug met **Shift+Tab**. Gaat de focus in omgekeerde volgorde?
4. Gebruik **pijltoetsen** binnen complexe widgets (tabs, menu's, radio buttons).
5. Activeer knoppen en links met **Enter** of de **spatiebalk**.
6. Let extra op dialoogvensters, lightboxes, mobiele menu's en cookiebanners: krijgen die direct de focus? Blijft de focus erin? Gaat de focus terug als je ze sluit?
7. Test de focusvolgorde op verschillende schermbreedtes. Responsive layouts kunnen de visuele volgorde veranderen, maar de HTML-volgorde (en dus de focusvolgorde) blijft hetzelfde.

### Voor developers

1. Tab door de pagina en vergelijk de focusvolgorde met de visuele volgorde. Als die sterk afwijken, check dan of CSS `order`, `flex-direction: row-reverse`, `grid` of `position: absolute` de visuele layout verandert zonder de HTML aan te passen.
2. Zoek in je code naar `tabindex` met positieve waarden (`tabindex="1"`, `tabindex="2"`, etc.). Verwijder ze en los de volgorde op in de HTML.
3. Test dialoogvensters: open elk venster en controleer of de focus erin verplaatst wordt, erin gevangen blijft (focus trap), en na sluiten terugkeert naar het trigger-element.
4. Controleer accordions en tabs: staat de content van een geopend paneel direct na de bijbehorende knop in de HTML?
5. Controleer skiplinks: navigeer met Tab naar de skiplink, activeer hem met Enter, en druk nogmaals op Tab. Belandt de focus in de hoofdcontent?
6. Test formuliervalidatie: verzend een formulier met fouten en check of de focus naar de eerste foutmelding verplaatst.
7. Gebruik de bookmarklet van [pauljadam.com/bookmarklets/focus.html](https://pauljadam.com/bookmarklets/focus.html) om de focusvolgorde visueel zichtbaar te maken.
8. Gebruik axe DevTools voor een eerste scan. Maar focusvolgorde-problemen zijn zelden automatisch te detecteren — je moet handmatig testen.
